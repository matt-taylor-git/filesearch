## Challenges and Learnings

### Requirements Gaps Discovered During Implementation
**Issue:** Story 2.3 required significant rework when network path support and auto-complete requirements emerged during implementation.

**Root Cause:**
- Missing environment requirements checklist during story drafting
- Assumptions about "typical directories" varied between team members
- Enterprise use cases not explicitly considered during planning

**Impact:**
- Lost ~1 day of development time on Story 2.3
- Required additional review cycles
- Created frustration for junior developer

**Lesson Learned:**
Need explicit environment/requirements checklist during story drafting that includes:
- Network path support requirements
- Enterprise environment considerations
- Edge case scenarios
- Integration points with existing systems

### Review Feedback Pattern
**Issue:** 3 out of 6 stories required significant review feedback and rework.

**Root Cause:**
- Stories under-specified during drafting
- No categorization system for review feedback priority
- Test scenarios written after implementation, not before
- Technical debt items deferred without scheduled resolution

**Impact:**
- Rework required mid-implementation
- Review cycles took longer than necessary
- Technical debt accumulated without plan

**Lesson Learned:**
Need systematic approach to review feedback:
- Categorize feedback as must-fix, should-fix, or enhancement
- Write test scenarios alongside ACs during story drafting
- Schedule technical debt resolution within 2 sprints
- Define "done" more comprehensively during story creation

### Technical Debt Management
**Issue:** 3 technical debt items identified but not scheduled for resolution.

**Debt Items:**
1. Auto-complete for common paths (home, documents, desktop)
2. Network path testing and documentation
3. Duplicate PyQt6 imports cleanup

**Root Cause:**
- No process for tracking and scheduling debt resolution
- "Later" became indefinite deferral
- No ownership assigned to debt items

**Impact:**
- Debt may accumulate and slow future development
- Missing functionality that users may expect
- Code quality slowly degrading

**Lesson Learned:**
Need explicit technical debt tracking with:
- Clear ownership assignment
- Scheduled resolution within 2 sprints
- Priority classification (high/medium/low)
- Regular review in standups
